## Q2. 転職理由を教えてください

### 結論

> 「フロントエンドエンジニアとして5年のキャリアを積み、技術力・マネジメント経験・顧客折衝の経験を一通り積んできた今、これを土台に**デザインエンジニア・ディレクション・PMなど、上流から事業に携われる環境**へステップアップしたいため」

### 理由（前向きに表現）

1. フロントエンドを軸にデザインと事業の両面へキャリアを広げたい
2. デザイナーとエンジニアが密に協業する環境で、UI / UX を起点に価値を生み出す動き方を磨きたい
3. リーダー経験を踏まえ、技術選定・設計判断により裁量を持って関わりたい

### 具体例（深掘りされた場なぜ当社を志望したのか合）

- 在籍していた受託企業では「納品して終わり」になりやすく、デザインの意図や運用後の改善まで踏み込めない案件もあった
- 直近の自社コーポレートサイトは、UI 設計 → 実装 → 運用フロー整備までを独力で完結させた経験
- 海外向け衣服レンタル案件で AI ツールを取り入れた経験から、デザインと実装の往復をさらに高速化できる環境に魅力を感じている

### 注意点

- 現職や前職への不満（待遇・人間関係など）は **絶対に言わない**
- 前職の会社都合移籍については聞かれたら簡潔に答える（→ Q2a 参照）
- 「現職では実現できないのか？」と聞かれたら → 「現職でもデザイン領域に踏み込む努力はしているが、在籍している企業ではクライアントワーク中心のため上流から継続的に関わる経験が積みにくい」
- **「環境を変えたい」だけで終わらせない**：受託 → 自社サービスへの転職は「不満からの逃避」に聞こえやすい。「自社サービスだから*できること*に挑戦したい」という表現に変える。例：「受託だとクライアントの要件通りに動くことが多く、PMやディレクションの経験を積みにくい環境でした。自社サービスであれば企画から運営まで一貫して関われるので、コードを書くだけでなくサービスをどう育てるかという視点も身につけたいと考えています」

## Q2a. 前職（80&Company）からの移籍について

### 結論

> 「会社都合による新会社設立に伴い移籍しました。事業内容・メンバーは引き続き同じで、実態としては変化なく業務を継続しています」

### 回答例

```
受託開発をしていた会社で財政状況が悪化したため、既存メンバー全員が新会社に移籍することになりました。
事業内容はすべて引き継いでいるため、実態としては変化なく同じプロジェクトを継続しています。
今回が初めての自主的な転職活動になります。
```

### 注意点

- 詳細を掘り下げられても前職の事情は深く話さない（「会社都合の移籍です」と簡潔に）
- 「現職に不満があるのでは？」という印象を与えないよう、事実ベースで淡々と答える

## Q2b. なぜエンジニアを目指したのですか？

### 結論

> 「コロナ禍でキャリアを再考したことがきっかけで、兄がエンジニアとして安定して働いていたことに影響を受けてエンジニアの道を選びました。実際にやってみてコーディングの面白さを見つけ、そこからフロントエンドを軸に積み上げてきました」

### 回答例

```
コロナ禍でキャリアの出発点を再考したことがきっかけです。
内定先の状況が不安定になる中で、兄がエンジニアとして安定して働いていたことに影響を受け、
エンジニアの道を選びました。

最初は「手に職をつけたい」という実利的な動機でしたが、Web制作会社でコーディングを始めると、
デザインを実際に動くプロダクトとして実装できることへの面白さを感じるようになりました。
「見た目を作るだけでなく、それを動くサービスにできる」という点にやりがいを覚え、
そこからNext.jsやTypeScriptを学び、フロントエンドを軸としたキャリアを積み上げてきました。
```

### 注意点

- 最初の動機が消極的であっても、**実際にやってみて興味が深まった流れ**を必ず添える
- 「なぜフロントエンドを選んだのか」の深掘りには → 「デザインに近い領域で、見た目に対する自分のこだわりが活かせると感じた」

## Q3. 転職活動の軸 / 会社選びで大事にしていることは？

### 結論（3軸で構造化）

1. **デザインとエンジニアリングが近い距離**で動ける環境（デザイナーと密に協業 / Figma を中心に運用）
2. **Next.js / TypeScript を中核**にしつつ、技術選定の裁量がある環境
3. **フルリモート or 出社週1〜2** までの柔軟な働き方

### 補足

- 4軸目があれば「自社開発で長期的にプロダクトを磨ける」を加えてよい
- 軸を1つに絞れと言われたら **1（デザイン × 実装）** が本命
- スタートアップを受ける場合は MVV（ミッション・ビジョン・バリュー）への共感も触れる

## Q4. 3〜5年後にどうなっていたいですか？

### 結論

> **「デザインエンジニアとして、UI / UX を起点にプロダクト価値を引き上げられる存在になっていたい」**

### 1年目 / 3年目 / 5年目で具体化


| スパン     | イメージ                                                                             |
| ------- | -------------------------------------------------------------------------------- |
| **1年目** | 入社企業のフロントエンドにキャッチアップし、デザイン再現性とコンポーネント設計の品質で貢献。チーム横断のコミュニケーション役を担う                |
| **3年目** | フロントエンドのリードとして、技術選定・設計判断・新規メンバーのオンボーディングを主導。デザイナーと開発フローを最適化する役割                  |
| **5年目** | デザインエンジニアリング領域の旗振り役として、デザインシステムの設計・運用や UI / UX 改善のリードに踏み込む。マネジメントとプレイヤーを 5:5 で担う |


### 注意点

- 「マネジメント特化」「プレイヤー特化」のどちらかに振り切らない
- 突飛なビジョンより **地に足ついた具体性** が評価される（tipsの 04 参照）
- 入社企業の事業ドメインに合わせて言葉を調整する

## Q5. これまでの成功体験を1つ教えてください

### 回答文（本番用）— 結果先出し構成

> **①結論・定量成果（ここから入る）**
> 「管理画面の構築工数を半日〜1日からほぼゼロにした、npmパッケージ化のお話になります。」
>
> **②3つのポイント + 詳細確認**
> 「成果につながった点が3つあります。工数の削減・品質のばらつき解消・他プロジェクトへの横展開です。詳しくお話しさせていただいてもよいですか？」
>
> **③具体的な取り組み（詳細）**
> 「海外向けECサイトを構築する中で、管理画面のCRUD処理は共通化できるのではないかと考えまして、Next.jsでshadcnとradixUIを組み合わせてnpmパッケージとして構築しました。以前は管理画面1つ作るのに半日から1日かかっていましたが、今はAPIの繋ぎ込みのみで完成するので構築工数はほぼゼロになりました。コードレビューも管理画面部分はほぼなくなり、作業者による品質のばらつきも解消できました。その後他プロジェクトにも展開して現在3プロジェクトで導入されていて、社内の標準管理画面として使われています。」

### STAR で構成（管理画面共通化・npmパッケージ化）

| 項目            | 内容                                                                                                    |
| ------------- | ----------------------------------------------------------------------------------------------------- |
| **Situation** | 海外向けECサイト構築で管理画面を作成する際、都度作成していた管理画面の工数（半日〜1日）と作業者による品質のばらつきが課題だった                                  |
| **Task**      | 管理画面のCRUD処理を共通化し、工数削減と品質均一化を実現すること                                                                    |
| **Action**    | Next.jsでshadcnとradixUIを組み合わせてコンポーネント設計を行い、npmパッケージとして配布できる形に構築。ESM/CJS両対応・TypeScript型定義同梱・Storybookによるドキュメント化も実施 |
| **Result**    | 管理画面の構築工数がほぼゼロに。コードレビューの負担も管理画面部分はほぼなくなり、品質のばらつきを解消。その後3プロジェクトに展開され社内標準として運用中                        |

### 深掘り対策（設計で工夫した点）

> 「npmパッケージ化が初めてだったので、ライブラリ用のビルド設定を調べながら構築しました。使う人の環境によって動かないことがないようにESM/CJS両対応にし、TypeScriptを使っているメンバーも補完が効いて使いやすいように型情報も同梱しました。またフロント領域でない人も使えるよう、Storybookでコンポーネントの見た目や使い方がわかるドキュメントを整備しました。」

### 深掘り対策（社内への普及）

> 「今まで標準版がなかったので導入自体はスムーズにできました。その後もメンテナンスや新機能の追加を継続していて、変更履歴（CHANGELOG）を管理しながらv0.1.97まで更新を重ねています。」

> ⚠️ 注意：「私だけが優秀だった」という印象にしない。**仕組みを作ってチームに渡せた**という視点を必ず入れる。

## Q6. 失敗体験とそこからの学びを教えてください

> 「海外向けのECサイトで、UIUXがよくない状態でリリースしてしまったことです。デザインが日本標準のUIになっていて、カートフローに確認画面が3枚入っていて全体がとても長くなっていました。離脱率が高くなる懸念があったのでデザイナーさんに相談したのですが、感覚的に話してしまって納得いただけず、そのままリリースしました。アナリティクスで確認するとやはり離脱率が高くなっていました。この経験から、人を説得するにはデータやエビデンスが必要だと学びまして、以後は相談する際には必ず調査をしてから臨むようにしています。」

### STAR で構成

| 項目                | 内容                                                                              |
| ----------------- | ------------------------------------------------------------------------------- |
| **Situation**     | 海外向けレンタルECサイトの構築。完成したデザインが日本標準のUIになっており、カートフローに確認画面が3枚入っていた                     |
| **Task**          | 海外標準のUIUXに合わせた設計にすること                                                           |
| **Action（失敗の中身）** | 「確認画面が多すぎると離脱率が上がる」とデザイナーに相談したが、データや根拠なく感覚的に話してしまい納得を得られなかった。そのままリリースした        |
| **Result**        | アナリティクスで離脱率が高くなっていることを確認。クライアントへの納品物の品質が下がってしまった                                  |
| **学び**            | 人を説得するにはデータ・エビデンスが必要。以降は相談の際に必ず事前調査・実例・数値を準備してから臨むようにした。複数選択肢とメリデメを提示するようにもなった |

### 深掘り対策（その後活かせた場面）

> 「それ以降は、クライアントへの提案の際に実例を挙げてイメージしやすくしたり、複数の選択肢とメリット・デメリットを提示するようにしています。クライアントから『わかりやすい』と言っていただけることが増えました。」

### 注意点

- 「努力したが厳しかった」「組織変更で手放さざるを得なかった」など、**自分の責任範囲を超えた話**にしない（誠実さを見られる）
- 失敗を語る際は、必ず **学びと再発防止策** までセットで話す

## Q7. 長所と短所を教えてください

### 長所（十の要素）

> **「デザインから実装、運用フローまでを一貫して担えること」**

## Q8. 使ってきた技術の選定理由・現在の課題を教えてください

### メイン技術スタック（PREP）


| 技術               | 選定理由・好きなところ                                                                                    |
| ---------------- | ---------------------------------------------------------------------------------------------- |
| **Next.js**      | App Router 以降のサーバーコンポーネント・Streaming SSR で UX を作り込みやすい。Vercel との親和性が高く、個人〜中規模プロダクトまでデフォルト選択肢になる |
| **TypeScript**   | 型でドキュメント代わりになり、リファクタリング耐性が上がる。チーム開発では特にバグの早期発見につながる                                            |
| **Tailwind CSS** | デザイントークンを統一しやすく、Figma との対応が取りやすい。ユーティリティクラスでコンポーネントの粒度判断がチームで揃いやすい                             |
| **Shopify Plus** | Liquid テンプレートと Hydrogen / カスタムフロントの選択肢がある。EC 特有のビジネスロジック（在庫・決済・税）が標準化されており、要件追従が速い             |


### 現在の課題（自己理解）

- **状態管理ライブラリの選定経験**が React Context / Zustand / Jotai の比較レベルに留まっており、大規模プロダクトでの Redux 系運用は実務経験が浅い
- **テスト文化**は現職で限定的だったため、Vitest / Playwright / E2E まで含めた品質保証パイプラインを体系的に整えた経験が少ない
- いずれも次の現場で取り組みたい領域として明確に意識している

### より良い方法があるか

- 状態管理：サーバー状態は **TanStack Query**、クライアント状態は **Zustand** という分担が、現代 Next.js プロジェクトでは扱いやすい
- スタイリング：チームの規模・デザインシステムの成熟度に応じて Tailwind / CSS Modules / vanilla-extract を使い分けるのが現実的

## Q9. 設計・開発で大切にしている思想は？

### 結論

> **「ロジック層と UI 層を明確に分け、デザインの意図を細部まで再現すること」**

### 理由

仕様変更とデザイン変更は別軸で発生する。両者を分離しておくことで、片方の変更が他方に波及しにくくなり、再利用性と保守性が両立する。

### 具体例（イオンリテール案件）

- ディレクトリ構造を `features / components / hooks / services` で分離
- 色・アニメーション duration・opacity をデザイントークンとして共通化
- Storybook による可視化で、デザイナー・エンジニア・QA が同じ画面でレビュー

### 失敗例（裏面）

- 初期プロジェクトでは UI に直接 fetch を書いてしまい、テストしにくく改修コストが高くなった経験あり。以降は徹底的に分離するようになった

### 締め

> 「思想だけでなく、Storybook 導入や Figma × GitHub Issue 連携など、**チーム全員が同じ思想で動ける仕組み**まで落とし込むのが大事だと考えています」

## Q10. パフォーマンスや再レンダリング対策の経験を教えてください

> ⚠️ **注意**：「レンダリング」という質問は意図が2種類ある。**React のre-rendering最適化**と、**Next.js のレンダリング戦略（SSG/ISR/SSR/CSR）**。まず意図を確認し、または両方カバーして話す。

---

### ① React の re-rendering 最適化

**結論**：**「不要な再レンダリングは『そもそも発生させない設計』で防ぎ、最後の手段として `useMemo` / `useCallback` を使う」**

具体例：

1. **Server Components の活用**：Next.js App Router では、状態を持たない部分はサーバーコンポーネントに閉じ、クライアントへ送るバンドルサイズを削減
2. **コンポーネント粒度の見直し**：頻繁に再レンダリングされる箇所と静的な箇所を分割し、useState の責務を末端に寄せる
3. **`memo` / `useCallback` / `useMemo`**：プロファイリング（React DevTools Profiler）で **実測した上で** 適用。先に貼らない
4. **画像最適化**：`next/image` で遅延読み込み + WebP 変換、LCP の改善

---

### ② Next.js のレンダリング戦略（SSG / ISR / SSR / CSR）

**結論**：**「ページの性質（更新頻度・SEO要否・ユーザー固有データか）の3軸で使い分ける」**

判断軸：

| 戦略 | 使いどころ | 例 |
|------|-----------|-----|
| **SSG** | 更新頻度低・SEO必要 | 静的コンテンツ、企業紹介ページ |
| **ISR** | 定期更新あり・SEO必要 | 求人一覧、記事一覧 |
| **SSR** | リクエスト毎に内容が変わる・SEO必要 | 検索結果ページ |
| **CSR** | ユーザー固有データ・SEO不要 | マイページ、ダッシュボード |

App Router での考え方：

- **Server Components**：データ取得をサーバー側で完結させ、クライアントへ届くJS量を減らす → LCP・TTI の改善
- **Client Components**（`'use client'`）：インタラクション・状態管理が必要な箇所のみに限定する

---

### ページ速度改善の経験が少ない場合の答え方

受託開発など、改善まで踏み込む機会がなかった場合：

> 「受託開発の現場だったので継続的な改善サイクルを回す機会は少なかったですが、LighthouseやPageSpeed Insightsでスコアは確認していました。next/imageの自動最適化やnext/linkのprefetchなど、Next.jsの機能を活用したパフォーマンス対応は意識して取り組んでいます。自社サービスの環境でCore Web Vitalsの改善に継続的に取り組んでいきたいと考えています」

---

### 課題感

- Lighthouse / Web Vitals を継続観測する仕組みは現職では整備し切れていなかった
- 次の現場では Speed Curve や CrUX レポートを使った継続観測を整備したい

## Q11. 業務での AI 活用についてどう取り組んでいますか？

### 結論

> **「Claude / Cursor / Figma MCP を組み合わせて、デザインから実装までの往復を高速化している」**

### 具体例（海外向け衣服レンタル新規事業）

- **Cursor + Claude**：コンポーネント設計案や Tailwind クラスの提案、type 定義の自動生成
- **Figma MCP**：Figma のデザイン情報を直接 Claude に渡し、コンポーネント雛形を自動生成。デザイン → 実装の初期コーディング工数を削減
- **設計判断の壁打ち**：技術選定や状態管理の比較検討で、Claude にメリット・デメリットを整理させてレビュー

### 自分なりの考え

- AI が出すコードは **そのまま使わず、必ず自分で意図を理解した上で採用**
- AI に渡す前に **要件と制約を明文化** することで、結果的にチームへの説明資料にもなる
- AI 活用は「個人の生産性」より「チームでのナレッジ流通」に効くと感じている

## Q12. マネジメントスタイル / チーム運営で意識していることは？

### マネジメント経験（事実）

- 4名（30代メンバー2名・オフショアメンバー2名）のマネジメント経験
- デイリースクラムのファシリテーション、タスク管理、スケジュール調整、コードレビューを担当
- 人事評価・給与決定までは未経験

### 結論

> **「チーム全体の潤滑油として、仕組みで自走できる状態を作ることを意識している」**

### チームでの役割（回答文・本番用）

> 「チーム全体の潤滑油のような役割だと思っています。一度、外注デザインの納品形式の手違いで、XDで受け取る予定がイラストレーターになってしまいまして、10人弱のチームで最新デザインを同期的に確認できない状態になりました。週に1〜2回デグレが発生していて、チームの認知負荷も高くなっていました。そこで前々からfigmaを導入したかったので、イラストレーターのデザインをfigmaの1ファイルに集約して、デザイン管理を自分が担当しました。結果的にデグレがゼロになり、その後社内の標準がXDからfigmaに移行しました。チーム全体の認知負荷を下げることに貢献できたと思っています。」

### 深掘り対策（リーダーサポートの具体例）

> 「リーダーのサポートも積極的に行っていました。技術的に優れているリーダーでも言語化やコミュニケーションが得意でない場合があるので、そういう時にリーダーの意図を聞いてメンバーに言語化して伝えるようなサポートをしていました。またリーダーの業務が集中しがちなので、できることはありますかと声がけして少しでも負担を減らすようにしていました。」

### 大切にしていること

- 「指示する」より「**仕組みで自走できる状態を作る**」
- 心理的安全性を担保するため、ファシリテーションでは発言量の偏りを意識
- 部下からのフィードバックを受け止める素直さを失わない

## Q13. 業務外の自己研鑽について教えてください

### 結論

> **「課題を自分で発見し、実際に手を動かして技術検証まで完結させることを習慣にしている」**

### 具体例（美容室サイトリプレイス）— 詳細版

```text
行きつけの散髪屋さんのサイトが表示崩れを起こしていることに気づき、
オーナーに直接提案してリプレイスを受注しました。

課題を見つけた背景は、自分自身がお客さんとして使っていたからです。
「こんな状態で公開されているなら、他にも同じ悩みを持つ
スモールビジネスは多いはず」という仮説を持ちながら動きました。

既存のWordPress制作会社と直接やりとりして移行の段取りを進め、
Next.js + Contentful + Vercel + Resend のスタックで再構築しました。

結果として年間ランニング費用をドメイン代のみ（約1,500円）に削減し、
その後の運用保守もいただいています。

ここで意識したのは、「サイトを作る」だけでなく
「オーナーが自分でコンテンツを更新できる状態にする」ことでした。
CMSの構成設計、管理画面の使い方レクチャーまで含めて
完結させたことで、継続的な関係に繋がっています。
```

### なぜその課題を見つけたのか・何を実現したいのか（ポイント）

- **課題発見の経緯**：ユーザーとして体験の悪さに気づいた → エンジニアとして「自分なら直せる」と動いた
- **見えていた課題**：表示崩れ・古いデザイン
- **見えていなかった課題**：運用コスト・オーナーがCMSを使えない構造・依存ベンダーのロック
- **先を見据えた動き**：単なる表示修正ではなく「長期的に自走できる構成」を設計した
- **実現したかったこと**：オーナーの「困った」を解決し、継続的な信頼関係を作ること

### その他の自己研鑽

- **ポートフォリオサイト [naoto-okawa.com](https://naoto-okawa.com)**：Figma で設計 → Next.js で実装 → Vercel + Contentful + Resend で運用。実務で試したい技術の検証場として継続的に更新
- **GitHub Pinned Repositories** でアウトプットを公開（Findy のスキル偏差値71達成）
- **Figma の活用拡大**：UI デザインだけでなく、Figma MCP を介した実装連携を試行
- **AI ツールのキャッチアップ**：Claude / Cursor のリリースノートを継続的にフォロー

### 学び方

- 公式ドキュメント・Zenn・Findy のテックブログを中心に情報収集
- 試したい技術は **必ず個人プロダクトで手を動かす**（読むだけで終わらせない）

## 🎯 全質問共通：話し方のチェックリスト

### 話す前

- 結論ファースト（PREP法）で構造化されているか
- 具体例（**数値・定量成果・固有名詞**）が入っているか（定性だけで終わらない）
- 「御社でどう活かせるか」を最後に添えているか
- 声に出して練習したか（頭の中で読むだけでは本番で出てこない）

### 話している最中

- 1問あたり **2分以内** に収まっているか（長すぎると相手が飽きる）
- 専門用語を多用していないか（採用担当が非エンジニアの場合あり）
- 言葉にネガティブさが滲んでいないか

### 話した後

- 相手に「で、それで御社では...」と続きを促せているか
- 緊張で早口になっていないか

## 🔺 ピラミッド構造での話し方（背景・成果が長くなる質問向け）

成功体験・設計の話・自己研鑽など、**話が長くなりそうな質問はピラミッド構造**で話すと
ロジカルで伝わりやすくなる。

```
① 結論 + 定量成果（一番短く・一番印象的に）
   例：「1ページあたりの実装時間を約50%短縮しました」

② ポイントを簡潔に（3つ以内に絞る）
   例：「3つポイントがあって、①設計の分離、②Storybook導入、③フロー整備です」
   
   → ここで一言確認を入れる：
   「詳しくお話しさせていただいてもよいですか？」
   （会話のラリーができる人という印象を与え、一方的にならない）

③ 詳細を話す（ここで背景・課題・行動・結果を展開）
```

> ✅ この「②→確認→③」の流れで、**聞き手をコントロールしながら話せる人**という印象を与えられる。
